# 一切皆颜色 · AuraRide Manifesto

> 这是 AuraRide 的**产品哲学**(manifesto),不是产品需求(PRD)、也不是接口契约(spec)。
> manifesto 描述"为什么这样"——它一旦立住就是不变量,反过来约束 [`PRD.md`](./PRD.md)、[`产品骨架-信息架构.md`](./产品骨架-信息架构.md) 和代码三层。
> 任何与本文档冲突的需求/特性/重构,先停下来 challenge 本文档,而不是直接改实现。

---

## 0. 一句话

> **像一部电影最终被人记成一个主色调,AuraRide 把一切都翻译成颜色。**

想想《布达佩斯大饭店》——你脑海里第一个浮现的是粉红;《疯狂的麦克斯》——是橙黄;《阿凡达》——是蓝绿。一部电影有十万帧、千万种光影,人记住的是**一个色**。这是大脑替我们做的压缩,丢掉细节,保住情绪。

AuraRide 把这种压缩做成产品的**主算法**:心情、路段、路线、照片、运动数据(心率/配速)、地点周围的店铺、网友的相册——形态各异,统一被**一个 5 分类的颜色空间**收口,使**导航、回顾、分享、社区**全部用同一种语言表达。

这是产品**唯一**的主线。其他都是它的副本与衍生。

> 💡 **给同学补一个 CS 背景的类比**:这种"万物归一接口"的设计,在 Unix 操作系统里就是著名的"一切皆文件"——把硬盘、键盘、网络连接都伪装成可以 `读 / 写` 的文件。我们这里是把万物伪装成"可以被染色的东西"。哲学相通,实现层是另一回事。

## 1. 五种颜色(当前色卡)

颜色 ID 是产品的"主键"。**任何对色卡的增减、改名、改 hex,都必须先改本文档,再镜像到 [`src/app/lib/moodColor.ts`](../../src/app/lib/moodColor.ts) 的 `ColorId` 枚举与 `COLOR_PROFILES`。**

| ID | 中文 | EN | Hex | 关键意象 |
|---|---|---|---|---|
| `calm-green` | 暗绿 | MOSS | `#34E89E` | 顺应风向,把心跳交还给潮汐 |
| `lonely-blue` | 深蓝 | DEPTH | `#4FA8FF` | 潜入暗流,把喧嚣沉降于底面 |
| `explore-yellow` | 赭黄 | TRACE | `#FFB54A` | 从容探索街区肌理,收集长长的回声 |
| `release-red` | 余火 | EMBER | `#FF3344` | 撕开风阻,让不安在直道上彻底燃尽 |
| `tired-gray` | 灰白 | VOID | `#C9D2D8` | 隐入网格,做一阵没有轨迹的风 |

**留扩展槽,不预先扩。** 未来可能新加(例:夜的紫、雾的青、雪的白),但**必须满足两个条件**才能进:(1) 一个真实的、已有真实用户的情感原型;(2) 至少 30% 的现有功能(地图、广场、相册、年度回顾)在它出现后会变得更清楚而不是更糊。否则就是色卡通胀。

## 2. 三层结构:点 → 段 → 线

世界是混沌的连续体。我们把它**离散化成三个层级**,每层只承担它该承担的事。

```mermaid
flowchart TB
    subgraph Route["🚲 路线 Route<br/>一次完整骑行 / 一条可分享路径"]
      direction TB
      subgraph S1["路段 Segment 1"]
        P1a[📷 照片] -.投票.-> S1
        P1b[📷 照片] -.投票.-> S1
        P1c[📷 照片] -.投票.-> S1
      end
      subgraph S2["路段 Segment 2"]
        P2a[📷 照片] -.投票.-> S2
        P2b[📷 照片] -.投票.-> S2
      end
      subgraph S3["路段 Segment 3"]
        P3a[📷 照片] -.投票.-> S3
        P3b[📷 照片] -.投票.-> S3
        P3c[📷 照片] -.投票.-> S3
      end
      S1 -.按长度加权投票.-> Route
      S2 -.按长度加权投票.-> Route
      S3 -.按长度加权投票.-> Route
    end

    classDef seg fill:#FFD27A,stroke:#FF9A2E,color:#000;
    classDef photo fill:#fff,stroke:#999,color:#000;
    class S1,S2,S3 seg;
    class P1a,P1b,P1c,P2a,P2b,P2c,P3a,P3b,P3c photo;
```

> 📖 **上图怎么看**:照片是最小单位,几张照片**投票**决定它所在路段的颜色;几段路再按长度**加权投票**,决定整条路线的代表色。颜色因此**从下往上"长"出来**,而不是从上往下硬塞。

| 层 | 颗粒度 | 几何 | 谁产生 | 谁染色 |
|---|---|---|---|---|
| **照片 Photo** | 点 | (lng, lat) + 时间 | 用户拍 / API 拉 / Web 搜 | 颜色分类器(§3) |
| **路段 Segment** | 线 | 两个路口端点构成的折线 | 高德路网切分 | 段上所有照片**投票** |
| **路线 Route** | 集合 | 一串路段 | 推荐算法 / 用户完成的骑行 | 段颜色**加权投票**(§4) |

**为什么是这三层?**
- 照片是**唯一带绝对位置的真实证据**——它把"这条路是什么颜色"从主观主张变成可观察的多源信号。
- 路段是**世界给我们的天然颗粒度**——比"每 50m 切一刀"自然,比"整条路"细致。它让"五彩斑斓的曲线"在屏幕上不卡顿、不琐碎。
- 路线是**用户能记住、能分享、能再走一次**的最小单位——颜色聚合到这一层才有传播价值。

## 3. 颜色分类器:一切多模态输入 → 5 类颜色

> 📖 **本节出现的术语**:
> - **LLM**(Large Language Model,大语言模型):像 ChatGPT、豆包、通义千问、DeepSeek 这种"读字、写字"的 AI。
> - **VLM**(Vision-Language Model,视觉语言模型):**LLM 的升级款,既能看图又能读字**,可以输入一张照片然后用文字回答"这张照片是什么场景、什么情绪"。豆包视觉版、通义千问 VL、GPT-4V、Gemini 都是。
> - **"多模态输入"**:技术黑话,意思是"啥都能塞进来"——文字、图片、心率数字、地理坐标,统统当成有效输入。
> - **"纯函数"**:工程上的说法,指"同样的输入永远给同样的输出,不偷偷读外部状态、不偷偷写外部状态",这样它好测试、好替换、好缓存。

颜色分类器是一个**统一签名**的纯函数(它的实现可以是 VLM、可以是查表、可以是平均色,但签名不变):

```ts
// 形式化定义(实现在 src/app/lib/colorEngines/ 下)
type ColorVote = { colorId: ColorId; confidence: number /* 0..1 */; reason?: string };
type ColorClassifier<Input> = (input: Input) => ColorVote;
```

**已实现的分类器(MVP):**

| 分类器 | 输入 | 实现路径 | 当前状态 |
|---|---|---|---|
| 心情文本 → 色 | 一句话中文 | [`lib/moodColor.ts`](../../src/app/lib/moodColor.ts) `analyzeMood` | ✅ 关键词命中(可平滑升级为 LLM/情感 API,签名不变) |

**规划中的分类器(填充顺序按"对路线染色贡献度"排):**

| 分类器 | 输入 | 落地形态 | 备注 |
|---|---|---|---|
| 照片 → 色 (用户上传) | 图像 + 位置 | VLM 调用(豆包/通义千问 VL/GPT-4V) | 主信号:用户的真实视觉 |
| 照片 → 色 (回退) | 图像 | 平均色 + HSL 归类 | VLM 不可用时的兜底,坦白知道效果略差 |
| 路线名+历史 → 色 | 字符串 + WebSearch 结果 | LLM 5 分类 | 解决"无人拍过的新路线"冷启动 |
| POI 文本 → 色 | 高德/大众点评返回的店名、品类 | 关键词加权(可换 LLM) | "夜市/居酒屋/酒吧" → release-red,"图书馆/咖啡馆/公园" → calm-green |
| POI 图片 → 色 | 大众点评 / 高德拉到的图 | 同"照片 → 色" | 比用户照片更不准,但密度更高 |
| 运动数据 → 色 | 心率/配速/海拔变化 | 阈值规则(高心率高变化 → release-red;低稳定 → calm-green) | 决赛阶段接 HealthKit 后激活 |

**核心规则:**
1. **所有分类器输出同一个 `ColorVote` 类型**——这样上层聚合不需要知道下游是 VLM 还是查表。
2. **所有分类器都坦白承认 lossy**——返回 `confidence` < 1,允许上层加权。
3. **新加分类器 = 在 `src/app/lib/colorEngines/` 下加一个文件**,不允许在调用点写一坨 if/switch。

## 4. 投票与聚合:从点到段到线

**段的颜色 = 段上所有照片投票的众数。**

```
segmentColor(segment) =
  argmax over colorId of
    Σ photos in segment   vote_p.confidence × source_weight(vote_p.source)
```

> 📖 **公式翻译给非数学背景的读者**:这是一句中文翻译——"把这段路上每张照片对它认为是什么颜色的'投票'乘以这张照片的可信度,再乘以这张照片的来源权重,然后看哪个颜色得分最高,就用那个颜色"。`Σ` 是"加起来"的意思,`argmax` 是"得分最高的那个赢"的意思,公式就是大白话的密码版而已。

其中 `source_weight`(待调优):
- 用户上传照片: 1.0(最真实)
- POI 图片: 0.5(地点对,但不是骑行视角)
- 路线名+历史: 0.3(粗粒度先验)
- 平均色回退: 0.2(最弱信号)

**路线的颜色有两种视图,都由"投票"产生:**

| 视图 | 算法 | 适用场景 |
|---|---|---|
| **五彩斑斓** | 每段保留自己的颜色,整条路线 = 一串异色折线 | 路线详情页:展示细节、强调"路是有起伏的" |
| **路线主色** | 各段颜色按**段长度加权**投票,赢家 = 路线代表色 | 首页路线卡片 / 广场缩略 / 分享卡 / 按色筛选 |

> 我们深知 5 分类必然损耗细节。但**这种损耗就是产品**:它把"难以言说的一段路"压扁成"可被记住、可被搜索、可被分享"的一种颜色,这是用户回来的钩子,也是 PRD §8 列的差异化叙事的根。如果有一天我们再做更高维的"颜色光谱条",它依然要能**坍缩**回 5 类——否则那个高维信号在产品的其他平面上(首页筛选、广场分类、年度色卡)没有立足之地。

**多次骑行的聚合(周/月/年代表色)** 沿用同一套思想,具体公式见 [`产品骨架-信息架构.md` §5](./产品骨架-信息架构.md#5-色彩聚合定义卖点的算法要可信也要好看)。这里不重复——只声明它是同一个原则的不同时间尺度。

## 5. 颜色作为产品的"语法":UX 怎么吃这个抽象

颜色 ≠ 装饰。它是**导航、筛选、回顾、分享**的语言。下表把所有 UX 入口对齐到颜色协议:

| 入口 | 颜色在这里的角色 | 现有实现 |
|---|---|---|
| 首页色轮 | 五个颜色 = 五个滑动主题 | [`pages/CyclingStart`](../../src/app/pages/cycling/) ✅ |
| 写心情 | 文本 → 色(分类器) | [`pages/MoodEntry`](../../src/app/pages/MoodEntry.tsx) ✅ |
| 颜色揭晓 | "为什么是这个色"的高光时刻 | [`pages/ColorReveal`](../../src/app/pages/ColorReveal.tsx) ✅ |
| 路线推荐 | 路线用其代表色绘制,选中加深 | [`pages/RouteMatching`](../../src/app/pages/RouteMatching.tsx) ✅ |
| 骑行 HUD | 主色 = 此次情绪色,轨迹同色 | [`pages/EnhancedRidingHUD`](../../src/app/pages/EnhancedRidingHUD.tsx) ✅ |
| 我的旅程·按颜色 | 用颜色当相册分类 | [`pages/Journal`](../../src/app/pages/Journal.tsx) ✅ |
| 周期总结·情绪光谱 | 色谱 = 色彩占比可视化 | [`pages/Overview`](../../src/app/pages/Overview.tsx) ✅ |
| 年度回顾·色彩指纹 | 全年色彩按占比编织 | 🟡 规划 |
| 广场 Discover | 按路线主色筛选 | [`pages/Discover`](../../src/app/pages/Discover.tsx) ✅ |
| **颜色索引相册**(新) | 按"颜色"看跨路线的图片集,点开任一张可反查"它所在路线" | 🔵 新增,见 §6 |
| 分享卡 | 卡片主视觉 = 主色,背面 = 色谱 | [`pages/ShareCard`](../../src/app/pages/ShareCard.tsx) ✅ |

**底层一致性:首页五色滑动 ≡ 路线五色筛选 ≡ 相册五色分组。** 任何一个入口若不能用颜色描述,就是这个入口和产品哲学错位了——要么改入口,要么 challenge 哲学,不要在中间打补丁。

## 6. 颜色索引相册:照片 ↔ 路线 的双向门

广场目前按"路线主色"分组(线性视图)。manifesto 主张再加一种**正交视图**:按"照片颜色"分组(点视图)。

```
路线视图(已有):  按 route.dominantColor 分 → 看路线 → 进入路线后才看到照片
照片视图(新):    按 photo.colorId 分 → 跨路线汇集 → 选任一张 → 反查它所属的路段 → 再看那条路线的全量照片(回到"五彩斑斓"视图)
```

**为什么这个视图非加不可:**
- 它直接验证了 manifesto 的双向性——颜色不只是"路线 → 颜色"的下游标签,也是"颜色 → 任何东西"的上游索引。
- 它让单张照片成为入口:用户可以从"我想看绿色的回忆"出发,而不是从"我走过哪些路"出发。
- 它让"投票机制"对用户可见:点进一张绿照片所在的路线,看到"绿色赢了 7 票、蓝色 3 票、其他 1 票",这种透明度本身就是说服力。

工程上这只是给现有 photos 表加一个 `colorId` 索引 + 一个新的 Journal/Discover tab,不需要新表。

## 7. 工程不变量(违反它的 PR 应被打回)

> ⚠️ **这一节是写给开发同学和 AI agent 看的**,设计同学扫一眼就好,不必深究细节。核心精神是:"五个颜色"这个契约是产品的脊椎,任何代码上的改动都不能让色卡和现实脱节。

**对 Claude Code / Codex / 任何 coding agent:** 下面这些是不可商量的,改它们之前先开 issue challenge manifesto。

1. **`ColorId` 单一真相源:** [`src/app/lib/moodColor.ts`](../../src/app/lib/moodColor.ts) 的 `ColorId` 类型与本文档 §1 的色卡表必须**逐字镜像**。新增颜色 = 改 manifesto + 改 ColorId + 全局搜索补全所有 `switch` 类型穷尽。
2. **颜色分类器统一签名:** 任何"新东西能染色"的能力,落到 `src/app/lib/colorEngines/<name>.ts`,导出 `ColorClassifier<T>`。**禁止在页面里写 if/switch 临时染色逻辑**——那是抽象泄漏。
3. **聚合策略集中:** 段聚合、路线聚合、周/月/年聚合,全部走 `src/app/lib/colorAggregate.ts`(规划中,见 §6 任务)。页面只调,不重写。
4. **5 分类是默认契约:** 任何返回"颜色"的函数,默认输出 `ColorId`(枚举),而不是 `hex`(字符串)。`hex` 只是渲染时通过 `COLOR_PROFILES[id].hex` 查出来的。这样改色卡时不会有遗漏的硬编码 hex。
5. **回退一定可见:** 分类器没拿到信号 → 返回 `tired-gray` 且 `confidence: 0.1`,UI 显示"未染色/灰白"。**禁止静默返回随机色或默认其他色**——那会让"灰白 = 空缺"的产品语义崩塌。
6. **lossy 是产品,不是 bug:** 任何"为了精度把 5 分类升到 256 类/HSL 连续空间"的 PR,先回去读 §4 最后一段。如需做精细信号(例:年度色谱),走"高维信号 + 默认坍缩回 5 类"的双轨,不许把 5 类拆掉。

## 8. 它不是什么(免误解)

- **不是色卡系统/设计 token。** 那是 [`default_shadcn_theme.css`](../../default_shadcn_theme.css) 的事。manifesto 关心的是**语义**层颜色,不是 UI 框架的边框灰阶。
- **不是情感分析模型。** 心情 → 色的分类器是 manifesto 的一个**实例**,不是 manifesto 本身。换成 LLM 之后 manifesto 不变。
- **不是 Strava 的"心率区间染色"。** 那是把一个连续标量映射到颜色梯度。我们是把**多模态混沌输入**收口到 5 个**有情感名字**的分类,完全不同。
- **不是必须 5 个、永远 5 个。** §1 说了可扩展,只是有门槛。
