# AuraRide 社交化 · 开发任务拆分（给程序同学）

> 目标：把现在的**纯本地单机** App 演进为**有社交功能**的产品。
> 核心原则：**所有数据访问已收敛到一个接口 `RideRepo`（`apps/web/src/app/lib/rideRepo.ts`）。**
> 前端页面只调 `repo.*`，永远不直接碰 localStorage / 后端 SDK。
> 你的工作 = **实现一个 `supabaseRepo`（或自建 API 的 repo），满足同一个接口，然后把 `rideRepo.ts` 末尾的 `export const repo = localRepo` 换成你的实现。零页面改动。**

---

## 0. 现状 & 契约

- 现在 `repo = localRepo`：rides 存 localStorage，社区(广场)用**本地种子数据**撑演示。
- 你要做的就是把 `localRepo` 里的方法逐个换成真实后端调用，**接口签名不许改**：

```ts
interface RideRepo {
  // 个人骑行日记
  listRides(): Promise<RideRecord[]>
  getRide(id): Promise<RideRecord | null>
  saveRide(ride): Promise<void>
  deleteRide(id): Promise<void>
  replaceAll(rides): Promise<void>
  // 账号
  currentUser(): Promise<PublicUser | null>
  // 社区 / 广场
  publishRide(rideId, {city, caption}): Promise<Post>
  isPublished(rideId): Promise<boolean>
  listFeed(filter?): Promise<Post[]>
  getPost(id): Promise<Post | null>
  toggleLike(postId): Promise<Post>
}
```

数据类型（`Post` / `PublicUser` / `FeedFilter` 等）都在 `rideRepo.ts` 里，直接复用。

## 1. 技术选型（建议）

| 项 | 建议 | 理由 |
|---|---|---|
| 后端 | **Supabase**（Postgres + Auth + Storage + Realtime + RLS） | 2 人队、免运维、含 Apple 登录 |
| 鉴权 | Supabase Auth：**Sign in with Apple** + 手机号 | 对齐 Apple 生态卖点 |
| 文件 | Supabase Storage（照片）→ 返回 CDN URL | 不再把 base64 塞进记录 |
| 原生 | Capacitor 插件：后台定位、相机、推送 | 决赛/上架阶段 |

## 2. 后端任务（Backend）

### B1. 数据库表（Postgres）
- [ ] `profiles`：`id(uuid, =auth.uid)`, `handle`, `avatar_color`, `dominant_color_id`, `bio`
- [ ] `rides`：现有 `RideRecord` 字段 + `user_id`, `visibility('private'|'public')`
- [ ] `photos`：`id`, `ride_id`, `url`, `color`, `caption`, `taken_at`（照片**不**进 rides 表）
- [ ] `posts`（广场）：`id`, `ride_id`, `user_id`, `city`, `caption`, `cover_color`, `route_shape(jsonb)`, `published_at`
- [ ] `likes`：`(user_id, post_id)` 唯一
- [ ] `follows`：`(follower_id, followee_id)`
- [ ] `comments`：`id`, `post_id`, `user_id`, `text`, `created_at`（Phase 3）

### B2. 行级权限（RLS，重点：情绪数据隐私）
- [ ] `rides`：本人可读写自己的；`visibility='public'` 才允许他人通过 `posts` 间接读
- [ ] `posts`：所有人可读；仅作者可写/删
- [ ] `likes`/`follows`：本人可增删自己的

### B3. 存储
- [ ] Storage bucket `ride-photos`，签名 URL；上传策略限本人目录
- [ ] 旧 base64 照片**一次性迁移脚本**：读 localStorage → 上传 Storage → 写 `photos.url`

### B4. 接口实现（= 写一个 `supabaseRepo`）
- [ ] rides 五个方法 → 查/写 `rides`(+`photos`)
- [ ] `currentUser` → `auth.getUser()` + `profiles`
- [ ] `publishRide` → 建 `posts` 行 + 置 `rides.visibility='public'`
- [ ] `listFeed(filter)` → 查 `posts` join `profiles`，支持按 `color_id` / `city` 筛、按时间/热度排
- [ ] `getPost` / `toggleLike` → 对应表
- [ ] 换掉 `rideRepo.ts` 末行 `export const repo = supabaseRepo`

### B5. （Phase 3+）
- [ ] 关注流 feed、评论、Realtime 订阅点赞数、通知、内容审核

## 4. 分阶段（对齐比赛节奏）

| 阶段 | 范围 | 后端 |
|---|---|---|
| **P0 初赛（现在）** | 分享卡片导出图 + 本地种子广场（演示形态） | ❌ |
| **P1** | 真账号 + 自己骑行云同步（换设备还在） | ✅ rides/profiles/auth |
| **P2** | 发布 + 广场只读（按颜色/城市逛、点赞） | ✅ posts/likes |
| **P3** | 关注 / 评论 / 通知 / Realtime | ✅ follows/comments |
| **P4** | 原生后台定位 + 推送 + 内容审核 + 上架 | ✅ Capacitor |

> **P0 完全不需要后端**——分享卡片 + 本地示例广场就能把社交故事讲圆（也是 PRD 的策略）。
> 你（程序同学）可以**并行**从 P1 的 Supabase 表 + `supabaseRepo` 开始,设计/前端继续打磨 UI,互不阻塞。
