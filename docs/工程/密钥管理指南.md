# 密钥管理指南

> **密钥单一真相源 = `AuraRide/AuraRide-env`**(私有 GitHub 仓库)。
> 本地约定 clone 到与主仓库**同级目录** `../AuraRide-env/`。
> 这份文件是约定的"为什么 + 怎么用",**具体操作**见 [`AuraRide-env/README.md`](https://github.com/AuraRide/AuraRide-env)。

## 设计动机(为什么搞这套)

| 反模式 | 问题 |
|---|---|
| 密钥散落在各自 `.env.local` | rotate 时漏掉;新成员上手难 |
| 密钥直接 commit 到主仓库 | 一旦泄露,history 里所有旧 key 都被读 |
| 全走 GitHub Secrets | CI 能用,**本地开发拿不到**(GH Secrets 故意只能写不能读) |
| 全走 SaaS(Doppler/腾讯 SSM) | 多账号体系、要钱、网络依赖 |
| 项目记忆 / 微信发文件 | 不可审计,版本控制不可见,易过期 |

## 我们选的:dotenvx public-key 加密 + 私有仓库

- **公钥**写在文件头(`.env.production` 顶部),公开,任何人能用它**加密新值**
- **私钥**在 `.env.keys`(gitignored,只在持有者本机),用来**解密**
- 这是非对称加密 → "**任何人可以投信进邮筒,但只有收件人能开**"
- 结果:仓库内容即便泄露,没有 `.env.keys` 也读不到任何明文

## 4 种密钥住哪里

| 类别 | 例子 | 存放 |
|---|---|---|
| **后端服务密钥**(真要保密) | DB 密码 / COS SecretKey / DashScope key / JWT secret | `AuraRide-env/.env.production`(密文,commit 进 git) |
| **前端公开 env**(本来就该公开) | `VITE_AMAP_KEY`(域名白名单) / `VITE_API_BASE_URL` | `AuraRide-env/.env.public`(明文,commit 进 git) |
| **CI 自动化用** | `DEPLOY_SSH_KEY` / `DOTENV_PRIVATE_KEY` | GitHub Actions Secrets(主仓库 settings) |
| **个人 ops 工作** | chenzhuowen 的 CAM SecretKey / 服务器 root 密码 | chenzhuowen 本机 + 项目记忆,不分发 |

## 给 AI agent(Claude / Codex / 任何)的强制规则

> ⚠️ **这一节是黄金法则的展开,违反就是 break build。**

### 规则 1:写 env 引用前,先在 AuraRide-env 登记

任何时候你写出以下代码:
```ts
import.meta.env.MY_NEW_VAR
process.env.MY_NEW_VAR
```
```go
os.Getenv("MY_NEW_VAR")
```
```python
os.environ["MY_NEW_VAR"]
```

如果 `MY_NEW_VAR` **不在** `../AuraRide-env/.env.production` 或 `../AuraRide-env/.env.public`,你必须:

1. 切到 `cd ../AuraRide-env/`
2. `git pull`(避免和别人冲突)
3. 决定 X 属于哪一档:
   - 真密钥 / 后端用 → `dotenvx set MY_NEW_VAR=PLACEHOLDER -f .env.production`
   - 前端公开 / build 进 bundle → 直接编辑 `.env.public`
4. `git add . && git commit -m "add: MY_NEW_VAR for <feature>"`
5. `git push`
6. **回到主仓库**,在 PR description 里说明:
   - "新增了密钥 `MY_NEW_VAR`,占位值 = PLACEHOLDER"
   - "chenzhuowen 需要在 [`AuraRide/AuraRide-env`](https://github.com/AuraRide/AuraRide-env) 里 set 真值后 rotate"

### 规则 2:禁止在主仓库出现明文密钥

任何形如:
```
DASHSCOPE_API_KEY=sk-xxxx
COS_SECRET_KEY="actual-secret-value"
postgres://user:realpassword@host
```

**不能**出现在:
- 任何 `.md` 文件(包括 docs/)
- 任何代码注释
- 任何 git commit message
- 任何 PR description
- 任何 issue
- 任何 README 示例(**示例值用 `<PLACEHOLDER>` 或 `xxx`,不要写真值即便是过期的**)

### 规则 3:看到 .env.keys 出现在不该出现的地方,立刻报警

`.env.keys` 应该**只**在以下位置出现:
- `~/Code/AuraRide-env/.env.keys`(本机持有者)
- GitHub Actions Secrets(以 `DOTENV_PRIVATE_KEY_PRODUCTION` 名义存)

任何其他位置(主仓库 / docs / commit / PR / 聊天) = **泄露**,立刻提醒 chenzhuowen 走 `scripts/rotate.sh`。

## 紧急情况:私钥泄露

如果 `.env.keys` 不小心被 commit / 发错 / 上传:

```bash
cd ~/Code/AuraRide-env
./scripts/rotate.sh   # 重新生成密钥对,所有密文重加密
# 新的 .env.keys 私聊发给所有持有者
# 旧的立刻作废
```

同时,所有真值密钥(DashScope / COS / JWT)应该在各自服务里 rotate 一次,因为旧 `.env.keys` 持有者能解开 history 里的旧密文。
