# ADR-003 · 后端技术栈 + 云架构 + 部署链路

> **状态**: 🟡 Proposed(待 chenzhuowen + 陈娟 review,review 后改为 ✅ Accepted)

> **作者**: chenzhuowen + Claude(基于 2026-06-20 ~ 06-21 6 小时对话沉淀)

## 背景

mvp-a 阶段陈娟的设计完成度已经 99%,但 4 个 lane 审计发现:**前端契约层 100% 就位、P0 demo 80% 完成、P1+ 真实后端 0%**。陈娟给 chenzhuowen 的任务是"以 `RideRepo` 接口为契约,含后端建表/RLS/存储/实现 + 前端 + P0→P4 分阶段 + 验收口径"。

同时,**初赛(2026-06-30)只交文档 + 海报,启明赛全程不要求可运行 App**,这意味着后端可以选择"赛后再启动"或"现在开始,demo 视频里用真闭环增强可信度"。chenzhuowen 选择**后者**——理由:

1. **作品说明文档"可行性分析"300 字**从"理论可行"升级为"已实现并部署在腾讯云",评委体感差距巨大
2. **复赛/演示视频**里"用户拍照 → VLM 染色 → 段颜色更新"如果是真闭环,1 秒胜过 30 秒演技
3. **设计 + 真数据反馈**让陈娟可以基于真实运行结果迭代视觉,而非靠想象

## 决定一览(每条都是这份 ADR 的"骨干")

1. 后端语言:**Go (API) + Python (Worker)** sibling 部署,非微服务
2. 数据库:**Postgres 16** 自托管
3. 缓存 + 队列:**Redis 7** 自托管
4. 反向代理:**Caddy 2**(可选但推荐)
5. 文件存储:**腾讯云 COS**(`auraride-photos-1315627382` @ ap-shanghai)
6. VLM(视觉语言模型):**阿里 DashScope 通义千问 VL**(跨云调用,国内稳定 + 支付宝)
7. 云宿主:**腾讯云轻量应用服务器**(`122.51.109.165`,2C4G6M,70GB SSD,Ubuntu 22.04)
8. 密钥管理:**dotenvx + AuraRide-env 私有仓库**(单一真相源)
9. 部署:**GitHub Actions → SSH → systemd**,生产服务器禁止开发
10. 架构形态:**模块化单体 + 异步 Worker**,**不上**微服务 / API Gateway / 服务网格

## 1. 语言选型:Go + Python sibling

### 决定

```
apps/api/      ← Go HTTP API (gin + gorm + JWT)
apps/worker/   ← Python 异步任务 (FastAPI + dramatiq + DashScope SDK)
```

### 为什么不是单一语言

| 方案 | 否决理由 |
|---|---|
| 纯 Go | VLM 调用 / 图像处理 / 异步任务编排在 Python 生态里 10 倍成熟(PIL / OpenCV / dramatiq / DashScope SDK 都是 Python first);硬塞 Go 写费时费力 |
| 纯 Python (FastAPI) | HTTP API 高并发 + WebSocket + 文件上传场景 Go 10 倍轻量;Python GIL / venv 部署成本高;对延迟敏感的 API 路径 Go 是合适工具 |
| 全 Node.js / Bun | 后端单语言全栈听上去美好,但缺 Go 的部署便利(单二进制)+ Python 的 ML 生态;两头不靠 |

### 为什么 sibling 而非 nested

不要 `apps/backend/{go,python}/` 这种按语言分的结构,理由:

- **`apps/*` 是"独立部署单元"**,Go API 和 Python Worker 是两个独立 systemd 服务,各自有 deploy 流水线
- **CI path filter 干净**:`apps/api/**` 触发 api 构建,`apps/worker/**` 触发 worker 构建
- **名字描述角色**(api / worker),不是描述实现细节(go / python)
- 未来加 Rust 服务,放 `apps/cdn-edge/` 与现有同级,**不动现有结构**

### 通信方式

```
浏览器 → Go API → (写 OSS + 写 Postgres + 推 Redis 队列)
                              ↓
                       Python Worker 消费队列
                              ↓
                       调 DashScope VLM
                              ↓
                       写回 Postgres 颜色字段
```

**Go 和 Python 通过 Redis 队列异步通信**,不直接 HTTP/gRPC 互调。共享同一个 Postgres,**这是模块化单体的标志,不是微服务**(微服务每服务独占数据库)。

## 2. 数据库 + 缓存

| 组件 | 选 | 自托管/托管 | 理由 |
|---|---|---|---|
| **Postgres 16** | apt 装在轻量服务器上 | 自托管 | RDS ¥120/月起,MVP 不值;轻量服务器 70GB SSD 装 Postgres 至少能撑到 5 万用户;后续真高峰再迁 RDS,`pg_dump + scp` 1 小时搞定 |
| **Redis 7** | apt 装在同台服务器上 | 自托管 | 队列 + session 缓存,~30MB RAM 占用,不值得 ¥60/月起的云 Redis |

### 我们为什么不上 RDS / 云 Redis

| 维度 | 自托管 | 云托管 |
|---|---|---|
| 月成本 | ¥0(已在轻量服务器内) | ¥120 + ¥60 = ¥180/月 |
| 备份 | 手写 cron pg_dump,scp 到 COS | 自动 7 天滚动 |
| 高可用 | 单机,坏了要修 | 主备自动切 |
| 性能调优 | 你自己 | 阿里/腾讯 DBA |
| MVP 阶段够用度 | ✅ 极够 | ✅✅ 有富余 |

**MVP 用自托管,P3+ 上线万级 DAU 后再评估**。`RideRepo` 抽象 + Postgres 连接字符串配置化,迁移时只改 `POSTGRES_URL`,代码 0 改动。

## 3. Caddy(可选反向代理)

### 决定

**用 Caddy 2**,但 Caddyfile 极简(~10 行):

```caddyfile
auraride.com {
    handle /api/* {
        reverse_proxy localhost:8080  # Go API
    }
    handle {
        root * /var/www/auraride
        try_files {path} /index.html
        file_server  # serve apps/web/dist
    }
}
```

### 为什么需要 Caddy

1. **自动 HTTPS**(Let's Encrypt 集成,5 行配置,自动续期)
2. **SPA `try_files` 兜底**(刷新非根路径回退 index.html,Go 自己写要多 20 行)
3. **将来加新 web 入口**(管理后台 / 监控面板)只多 4 行 Caddyfile,Go 自己路由要重写

### 为什么 Caddy 不是 API Gateway

Caddy 在这里只做"反向代理 + 静态文件 + HTTPS"——这是**Web Server** 工作,不是 **API Gateway** 工作。

| API Gateway 做的事 | 我们做不做 |
|---|---|
| 多服务路由(Kong/Zuul/Envoy) | ❌(我们就一个 Go API) |
| 中心化认证 / 鉴权 | ❌(Go 自己处理 JWT) |
| 限流 / 熔断 / 重试 | ❌(MVP 不需要) |
| API 版本管理 | ❌(单服务,路径里 `/v1/` 一处管) |
| 服务发现 | ❌(就一个 host,Caddy 反代到 localhost) |

我们的尺度还到不了 API Gateway 的位置,**预先架那一层 = 给自己挖一个不必要的复杂度坑**。

### 备选:不要 Caddy,纯 Go 一把梭

Go 1.21+ 有 `certmagic`,30 行代码实现自动 HTTPS。如果想极端简化,Go 自己处理 :443 + serve 静态 + 路由 /api/* 也可以。但**调试痛苦**——你 SSH 上去想改 SPA fallback,得重新 build Go binary;Caddyfile 直接 `caddy reload` 一秒。

**采用 Caddy** 是工程审美 + 长期可维护性的选择,不是必需性。

## 4. 文件存储:腾讯云 COS

### 决定

桶 `auraride-photos-1315627382` @ ap-shanghai(已建,已验证 coscmd 端到端 list/upload/delete)。

**铁律:照片永远走 OSS,不走 ECS**。流程:

```
浏览器 ──(请求上传 URL)──→ Go API
       ←(返回 STS 签名 URL)──
       ──(PUT 直传)─────────→ COS
                                   ↓
                              触发后端 webhook 或者
                              Go API 在响应中直接通知
                                   ↓
                              Python Worker 取这张图,调 VLM,写颜色
```

ECS 带宽 6M,但 OSS 带宽是独立的(走 COS 自带 CDN),**ECS 完全不感知图片流量**。

### 为什么 STS 临时签名 URL 而不是直接给前端 SecretKey

**SecretKey 泄露 = 我整个桶被删/被滥用**。STS 临时签名 URL:
- Go API 用 SecretKey 签 1 个有效期 10 分钟的 PUT URL
- 给浏览器
- 浏览器 PUT 到这个 URL,完事
- SecretKey 永远不离开服务器

详见后续 P1 工程时的 `apps/api/internal/upload/` 实现。

## 7. 密钥管理(dotenvx + AuraRide-env)

### 决定

**单一真相源 = `AuraRide/AuraRide-env`** 私有仓库,与主仓库同级 clone(`~/Code/AuraRide-env/`),用 dotenvx 非对称加密:

- **公钥写在 `.env.production` 文件头**(commit 进 git,任何人能用它加密新值)
- **私钥在 `.env.keys`**(NEVER commit,只在持有者本机和 GH Actions Secrets)
- **陈娟纯前端工作**只需 clone 这俩仓库 + `pnpm install` + `pnpm dev`,**不需要拿 `.env.keys`**

详见 [`docs/工程/密钥管理指南.md`](../工程/密钥管理指南.md)。

主仓库 [`AGENTS.md`](../../AGENTS.md) 黄金法则第 10 条强制约定:

> 任何写 `process.env.X` / `import.meta.env.X` / `os.Getenv("X")` 的代码,**X 必须先在 `../AuraRide-env/.env.production`(后端真密钥)或 `../AuraRide-env/.env.public`(前端公开 env)里登记**。

## 8. 部署:GH Actions → SSH → systemd

### 决定

**生产服务器 = 只跑二进制,禁止开发**。代码改动**必须**经过 git commit → push → GH Actions build → SSH 推送 → systemd restart 流程。

```yaml
# .github/workflows/api.yml(概念)
on:
  push:
    branches: [main, mvp-a]
    paths: ['apps/api/**', 'packages/types/**']
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-go@v5
        with: { go-version: '1.22' }
      - run: cd apps/api && go build -o api ./cmd/api
      - name: scp 到服务器
        run: scp apps/api/api deploy@$HOST:/opt/auraride/bin/api.new
      - name: 原子重启
        run: ssh deploy@$HOST 'mv /opt/auraride/bin/api{,.bak} && mv /opt/auraride/bin/api{.new,} && sudo systemctl restart auraride-api'
```

类似的 `worker.yml` 部署 Python worker(`pip install` 在服务器上跑 / 或者用 `uv` / 或者打 wheel)。

### 关键工程纪律

1. **deploy 专用 SSH 用户**(不是 root,不是 ubuntu),GH Secrets 里只放它的 key,权限只够"写 `/opt/auraride` + `systemctl restart` 它自己的 service"
2. **Atomic deploy**:新 binary 写到 `api.new`,success 后 rename;失败回滚一秒
3. **零停机非必要**(MVP 阶段):systemd 重启 1-2 秒中断接受
4. **env 文件不动**:GH Actions 部署时**不碰** `/etc/auraride/api.env`,systemd 重启时它继续从那里读

## 9. 架构形态:模块化单体 + 异步 Worker

### 决定

**模块化单体(modular monolith)+ 异步 worker** = 2 个进程,共享一个 Postgres,通过 Redis 队列解耦。

### 这不是微服务

| 维度 | 我们 | 微服务 |
|---|---|---|
| 服务数 | 2 (api + worker) | N 多 |
| 数据库 | 1 个共享 Postgres | 每服务独占 |
| 通信 | Redis 队列(异步) | HTTP/gRPC(同步) |
| 部署单元 | 1 台服务器 | k8s / 多机 |
| 服务发现 | 不需要 | Consul / Istio |
| 服务网格 | 不需要 | Linkerd / Istio |
| API Gateway | 不需要 | Kong / Envoy |

### 拆 Go / Python 不是按"服务"切,是按"语言生态"切

Go API 和 Python Worker **不是两个服务**,是**同一个产品的两个进程**,只是因为"HTTP 用 Go 写干净 + 异步 VLM 用 Python 写干净"才物理分开。共享数据库就是证据——真微服务这是禁忌。

### 何时该升级到微服务

- 多个独立产品线挤在一个 codebase
- 20+ 工程师并行开发,合并冲突频繁
- 不同服务 SLA 要求不同(支付服务 99.99%,日志服务 99%)

**我们都不是**。模块化单体能撑到 50 万 DAU(参考 Basecamp / Shopify 早期),AuraRide 这辈子都到不了那里。

## Caveats(我们当前不知道的)

1. **Postgres 自托管的备份策略还没定**——cron pg_dump 推 COS?推 GitHub release?推阿里云 OSS 备份?P1 阶段决定。
2. **Python worker 的并发数还没调优**——MVP 跑 1 个 worker 进程,P2 阶段视队列堆积情况调到 2-3。
3. **`apps/api/` 与 `apps/worker/` 共享的 Postgres schema 怎么管理**——sqlc / golang-migrate / Alembic?ADR 写到这里没结论,P1 启动时 spike 决定。
4. **腾讯云子账号 `auraride-claude` 的具体权限**——chenzhuowen 截图给的是第 1 页(10 条策略,看似没 COS Full Access),但 coscmd 实测能用,推测第 2 页有相关策略。chenzhuowen 需要补确认。
5. **DashScope 在国内的 token 限速 / 速率 / SLA**——MVP 阶段一定够用,但产品化时需要调研付费档位。
