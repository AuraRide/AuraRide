# ADR-005 · 后端先行 + 陈娟无感 + 工作流分配

> **状态**: 🟡 Proposed(待 chenzhuowen + 陈娟 review,review 后改为 ✅ Accepted)
> **日期**: 2026-06-21
> **作者**: chenzhuowen + Claude
> **依赖**: [`ADR-003-后端技术栈.md`](./ADR-003-后端技术栈.md) · [`ADR-004-monorepo重组.md`](./ADR-004-monorepo重组.md)

## 背景与动机

ADR-003 决定了后端用什么(Go + Python + Postgres + 等等),ADR-004 决定了仓库怎么放(`apps/web/api/worker`)。**但还有 3 个关键决策尚未纸面化**:

1. **后端 vs iOS 的执行次序** —— 谁先做?为什么?
2. **陈娟工作流的具体保护** —— 后端上线后她改 React 会不会被影响?如何"无感"?
3. **谁做什么 / 怎么协作** —— PR / Issue / Claude Code 自助 / 假数据 / Seed 策略怎么定?

这份 ADR 把这 3 件事说清楚,作为接下来 2-4 周开发节奏的总纲。

## 决定一览

1. **次序**:后端先(servers + API + Worker + Postgres + Redis + Caddy + COS + DashScope 接入)→ Capacitor 套 iOS(估算 1-3 天)
2. **陈娟保护**:web 通过 `RideRepo` 抽象层 + 新建 `apiRepo` + Vite proxy 透明转发,**陈娟代码 0 改动**,只 `pnpm install` 一次
3. **Seed 策略**:5 个假用户(2 真陈娟 + 你 + 3 虚拟骑友)+ 5 条上海预录路线 + 25-30 张真实街景图片 + VLM 染色预生成
4. **角色分工**:陈娟 = 设计 + 前端实现 + 哲学迭代;chenzhuowen = 后端 + 基础设施 + iOS 套壳 + 兜底
5. **新 feature 流程 2 选 1**:陈娟用 Claude Code 自助实现 + 合并;或陈娟提 Issue + chenzhuowen 实现
6. **server-as-CI**:一台服务器同时跑 prod(:443) + staging(:8443) + PR preview channels(:8001-8099)
7. **本机只编译前端**:两人开发机不跑 Postgres/Go/Python;后端走 CI 验收 + 自动部署

## 1. 为什么"后端先,iOS 后"

### 推理链

**目标**:复赛(7 月)演示视频里"用户拍照 → 颜色变化"是真闭环。

**这个目标拆开**:
- 闭环 = 拍照 → 上传 → VLM 染色 → 入库 → 前端拉新数据 → 路段颜色更新显示
- 6 步,**5 步在后端**,**1 步在 iOS 壳(拍照交互)**

**所以**:后端不通,iOS 套了壳也只是个"漂亮的 mock";后端通了,iOS 套壳只需 1-3 天(Capacitor `cap add ios && cap sync && xcodebuild`)。**真功夫在后端**。

### 时间预算(乐观估计)

| 阶段 | 工期 | 谁 |
|---|---|---|
| 服务器初始化(SSH 加固 + Postgres + Redis + Caddy + Go + Python) | 1 天 | chenzhuowen + Claude |
| 备案中(7-15 天 wall clock)期间后端开发 | 5-7 工作日 | chenzhuowen + Claude |
| Go API 骨架(auth + rides + posts + photos) | 3-5 天 | chenzhuowen + Claude |
| Python worker 骨架(VLM 染色 + 异步聚合) | 2-3 天 | chenzhuowen + Claude |
| 假用户 + 真照片 seed 数据 | 1 天 | chenzhuowen + 陈娟(选图)|
| iOS Capacitor 套壳 + 第一个 TestFlight | 1-3 天 | chenzhuowen + Claude(Apple ID 等审核) |

**总计 ~3 周(并行 + 备案期 + 开发)**,**6/30 初赛(只交文档+海报)不阻塞**,7 月初出第一版真后端 demo,**给复赛视频用**。

## 2. 陈娟工作流的"零感知"保护

### 当前现状(2026-06-21)

陈娟所有 14 个页面通过 `apps/web/src/app/lib/rideRepo.ts` 的接口访问数据:

```ts
// rideRepo.ts (现状)
export interface RideRepo {
  listRides(): Promise<RideRecord[]>
  getRide(id): Promise<RideRecord | null>
  saveRide(ride): Promise<void>
  // ... 11 个方法
}

const localRepo: RideRepo = { /* localStorage 实现 */ };
export const repo: RideRepo = localRepo;  // ← 切换点在这一行
```

### 改造方案(实施时)

新增 `apps/web/src/app/lib/apiRepo.ts`(我写,陈娟不看):

```ts
import type { RideRepo } from "./rideRepo";

export const apiRepo: RideRepo = {
  listRides: async () => {
    const r = await fetch("/api/rides");
    return r.json();
  },
  getRide: async (id) => {
    const r = await fetch(`/api/rides/${id}`);
    return r.ok ? r.json() : null;
  },
  saveRide: async (ride) => {
    await fetch("/api/rides", { method: "POST", body: JSON.stringify(ride), headers: {"Content-Type": "application/json"} });
  },
  // ... 11 个方法对应 fetch
};
```

改 `rideRepo.ts` 末行 1 处:

```diff
- export const repo: RideRepo = localRepo;
+ export const repo: RideRepo = apiRepo;
```

改 `apps/web/vite.config.ts` 加 5 行:

```diff
  export default defineConfig({
+   server: {
+     proxy: {
+       '/api': {
+         target: 'https://api.auraride.cn',  // 备案前用 IP:8080
+         changeOrigin: true,
+       },
+     },
+   },
    plugins: [react(), tailwindcss()],
  });
```

### 改造后陈娟工作日常

| 她做的事 | 改造前 | 改造后 | 差异 |
|---|---|---|---|
| 改 page UI | `pnpm dev` 跑 | 同上 | 0 |
| 加新 page | 写 + 注册 routes | 同上 | 0 |
| 测试数据 | 点"载入演示数据" | 自动有(后端 seed) | **少 1 个动作** |
| 改 manifesto §5 入口 | 改 markdown + 改 component | 同上 | 0 |
| 加新 RideRepo 方法 | 接口加 + localRepo 加 | 接口加 + apiRepo 加 + 通知 chenzhuowen 实现 server 端 | 多 1 步通知 |
| 改 color 染色逻辑 | 改 moodColor.ts | 同上 | 0 |
| 提交 PR | git push | 同上 | 0 |
| 看效果 | 浏览器 localhost:5173 | 同上(背后走 staging API) | 0 |

### 她为什么不会被后端崩溃阻塞

- **staging server 永远在线**(我和 chenzhuowen 维护,系统级 systemd 自动重启)
- **localhost dev 通过 Vite proxy 透明转 staging**,她本机不跑任何后端
- **极端兜底**:`apiRepo.ts` 的 fetch 失败可以 fallback 到 `localStorage` 的最后一次快照(类似 PWA 的离线缓存),她大部分页面继续可用

### `localRepo` 何去何从

**保留代码**(不删),但不再是默认 export。**变成"演示数据生成器" + "离线兜底"** 双角色:
- 测试时手动切换 export 可以快速回退到本地
- 网络断时 apiRepo 失败 → 自动降级读 localStorage 的缓存

## 5. 新 Feature 工作流(2 种,陈娟选)

### 流程 A:陈娟用 Claude Code 自助实现(更快,推荐 90% 时候用)

```
陈娟想加新功能 X(纯前端)
  ↓
她开 Claude Code,描述要做的事
  ↓
Claude 改 apps/web/ 下的代码,可能在 RideRepo 接口里加方法
  ↓
本地 pnpm dev 验证视觉效果 OK
  ↓
git push 到 feat/X 分支 + 开 PR
  ↓
CI 自动跑 typecheck + build,绿了她合并
  ↓
GH Actions 自动部署到 staging (随后 prod)
  ↓
如果加了 RideRepo 新方法 → chenzhuowen 收到通知,实现 server 端
```

### 流程 B:陈娟提 Issue + chenzhuowen 实现(后端重度时用)

```
陈娟开 GitHub Issue:"功能 Y 需要后端做 XXX"
  ↓
chenzhuowen 实现 server 端 + 推 apps/api
  ↓
CI 自动部署后端
  ↓
陈娟拉新代码,接入新 API(可能用 Claude Code 自助接)
  ↓
合并 + 自动部署
```

### 流程 C(应急):陈娟做前端 mock,先合并,后端后补

```
陈娟在 apiRepo.ts 里临时返回 mock 数据(不调 fetch)
  ↓
她的页面照常开发,看到完整数据
  ↓
合并到 main,staging 也跑得起来(用 mock)
  ↓
chenzhuowen 后续实现真 server,把 mock 删掉
```

这种**前端先行,后端补齐**的模式是行业标准做法,**陈娟掌握节奏**。

## 6. Server-as-CI:一台服务器跑 prod + staging + PR preview

详见 [`infra/caddy/Caddyfile`](../../infra/caddy/Caddyfile)。

```
122.51.109.165 (Caddy 监听 :443 + wildcard)
├── auraride.cn              → :8080 (prod web + api)
├── staging.auraride.cn      → :8443 (staging,陈娟和你的"latest mvp-a")
└── pr-N.preview.auraride.cn → :80NN (每个 PR 一个 channel,自动起灭)
```

PR 开/关时,GH Actions 调 SSH `infra/scripts/spawn-preview.sh N` 自动:
1. clone PR 分支 build
2. 起 systemd unit `auraride-pr-N.service` 监听 :80NN
3. Caddy 用 wildcard 证书自动 route
4. PR description 自动评论 preview URL
5. PR 关闭 → 调 `kill-preview.sh N` 自动清理

陈娟开 PR → 几分钟后她和你能点开 https://pr-42.preview.auraride.cn 看效果,**不用拉对方分支到本机**。

## 7. 本机只编译前端

| 你的电脑 | 陈娟的电脑 | 服务器 |
|---|---|---|
| Vite dev (web) | Vite dev (web) | Go API + Python Worker + Postgres + Redis + Caddy |
| 没有 Go | 没有 Go | Go 1.22 |
| 没有 Postgres | 没有 Postgres | Postgres 16 |
| 没有 Python(worker)| 没有 Python(worker) | Python 3.12 + dramatiq |
| 通过 Vite proxy 调 staging API | 同上 | 自跑 |

**本机:启动时间 < 5 秒**(Vite hot reload)
**服务器:24/7 在线**,我们 SSH 上去也只是看日志 / 改 systemd,不开发

## 8. Caveats / 未解决的事

1. **陈娟需不需要在 ADR-005 上签字?** 我建议**她在 GitHub UI 上 review 这个 PR 并 approve**,效果等同签字。
